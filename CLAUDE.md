# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev        # Start Vite dev server
npm run build      # tsc + vite build (type-check + bundle)
npm run preview    # Preview production build locally
```

There are no tests. Type-check only happens via `tsc` inside `npm run build`.

### Supabase Edge Functions

Edge Functions are written in Deno (TypeScript). Deploy individually:

```bash
npx supabase functions deploy verify-telegram
npx supabase functions deploy submit-claim
npx supabase functions deploy verify-bingo
npx supabase functions deploy call-number
npx supabase functions deploy generate-lobby-cards
```

### Database migrations

Migrations are plain SQL files run manually in the **Supabase Dashboard → SQL Editor**. They are not applied automatically. Run them in filename order when setting up a fresh project.

## Environment Variables

Required in `.env` (copy from `.env.example`):
- `VITE_SUPABASE_URL` — Supabase project URL
- `VITE_SUPABASE_ANON_KEY` — Supabase anon/public key

Required as Supabase Edge Function secrets (Dashboard → Edge Functions → Secrets):
- `TELEGRAM_BOT_TOKEN` — from BotFather; used only by `verify-telegram`
- `SUPABASE_SERVICE_ROLE_KEY` — auto-injected by Supabase into Edge Functions

## Architecture

### Frontend — Vite + React SPA on Vercel

Single-page app with React Router v6. No server-side rendering. `vercel.json` rewrites all paths to `index.html`.

The Supabase JS client (`src/lib/supabase.ts`) is initialized once as a module singleton with `createClient<any>`. All DB access goes through it.

### Backend — Supabase

- **Database**: PostgreSQL with RLS on every table
- **Realtime**: Postgres changes pushed to clients via `supabase_realtime` publication. Tables subscribed: `lobbies`, `called_numbers`, `bingo_claims`, `lobby_players`, `lobby_cards`, `game_history`
- **Auth**: Custom Telegram-based auth — no email/password UI. Players are identified by their Telegram ID. The `verify-telegram` Edge Function validates the Telegram HMAC, creates a Supabase auth user (email = `tg_{id}@bingo.local`, password = HMAC-derived), and returns a session token.

### RLS patterns

A critical `security definer` function `public.is_admin()` queries `players.is_admin` bypassing RLS — this prevents infinite recursion when other table policies check the `players` table. All admin-only policies call `public.is_admin()` instead of `exists (select 1 from players ...)`.

Player insert/update policies use `id::text = auth.uid()::text` (self-based). The `players.id` is always set to match the Supabase auth user UUID.

The `kick_player_false_claim(lobby_id, player_id)` function is a `security definer` RPC — it lets the `submit-claim` Edge Function kick a player using the player's own JWT without needing service-role bypass.

### Data model

| Table | Purpose |
|---|---|
| `players` | One row per Telegram user. `is_admin` flag set manually in Supabase dashboard. |
| `lobbies` | Created by admin. `stake_amount`, `status` (waiting → active → paused → completed), `prize_pool`. |
| `lobby_cards` | 400 pre-generated BINGO cards per lobby. `player_id` is null until claimed. Generated by `generate-lobby-cards` Edge Function. |
| `lobby_players` | Players who have claimed a card. Has `status` (joined/kicked), `false_claim_count`, `auto_clicker_enabled`. |
| `called_numbers` | Numbers drawn so far in a lobby. Unique per lobby+number. |
| `bingo_claims` | Submitted BINGO claims. `lobby_card_id` is the new FK (old `card_id` FK to `bingo_cards` is legacy/nullable). Status: pending → verified/rejected. |
| `game_history` | One row per completed game (unique constraint on `lobby_id`). Created by `verify-bingo` on approve. |
| `bingo_cards` | Legacy table — assigned when admin starts game. Superseded by `lobby_cards` for the new card-selection flow. |

### Edge Functions

All Edge Functions use `createClient(url, anonKey, { global: { headers: { Authorization: authHeader } } })` with the caller's JWT — **not service role** — so RLS policies apply normally. `SUPABASE_URL` and `SUPABASE_ANON_KEY` are auto-injected by Supabase.

| Function | Caller | Purpose |
|---|---|---|
| `verify-telegram` | Player (anon, via anon key) | HMAC validate Telegram initData → create/sign in auth user → return session |
| `submit-claim` | Player (session JWT) | Server-side bingo validation. False → RPC kick. Valid → insert pending claim. |
| `verify-bingo` | Admin (session JWT) | `action: approve` → complete lobby + game_history. `action: reject` → kick player, lobby continues. |
| `call-number` | Admin (session JWT) | Draw random uncalled number, insert into `called_numbers`. |
| `generate-lobby-cards` | Admin (session JWT) | Generate 400 bingo cards for a lobby in 4 batches of 100. |

### Claim flow (two-phase)

1. Player clicks BINGO → `submit-claim` Edge Function validates server-side:
   - **Invalid pattern**: `kick_player_false_claim` RPC → player status = kicked → lobby continues
   - **Valid pattern**: `bingo_claims` row inserted with status = pending
2. Admin sees pending claim in `/admin/game/:id` with card preview + detected patterns
3. Admin clicks **Verify Winner** → `verify-bingo action=approve` → lobby completed, `game_history` created
4. Admin clicks **Reject** → `verify-bingo action=reject` → player kicked, lobby continues

### Realtime subscriptions

Hooks subscribe to Postgres changes and update local React state:
- `useLobby(lobbyId)` — listens for `lobbies` UPDATE → navigates player when status changes
- `useCalledNumbers(lobbyId)` — listens for `called_numbers` INSERT → live number board
- `useLobbyCards(lobbyId)` — listens for `lobby_cards` UPDATE → card grid taken/available state
- `useBingoClaims(lobbyId)` — listens for `bingo_claims` changes → admin claims panel

`GameRoom` also has an inline subscription to `bingo_claims` filtered by `lobby_id` to detect when the player's own claim is verified or rejected (realtime kick feedback).

### Admin-side auto-calling

`AdminGame.tsx` runs `setInterval(() => callNumber(), 3000)` in a `useEffect` when `autoCall` toggle is ON and lobby status is `active`. The interval calls the `call-number` Edge Function. Numbers stop at 75. Auto-call state is local to the admin's browser tab — if the tab closes, calling stops.

### Bingo card format

Cards are stored as `(number | null)[][]` — a 5×5 grid where `card[row][col]` holds the number. `card[2][2]` is always `null` (FREE space). Columns follow standard BINGO ranges: B(1–15), I(16–30), N(31–45), G(46–60), O(61–75). Logic lives in `src/lib/bingo.ts`: `generateCard()`, `checkWin()`, `detectWinningPattern()`, `columnLetter()`.

### SuperAdmin setup

Set `is_admin = true` on a player row in the Supabase dashboard after that player has logged in once. There is no admin invite UI in V1.
